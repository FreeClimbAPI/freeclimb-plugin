# Commands Directory

## Layout

Command implementations live in this directory as oclif v4 TypeScript modules. Manual (non-generated,
non-executor) commands include `api.ts`, `describe.ts`, `diagnose.ts`, `login.ts`, `logout.ts`, `status.ts`,
`dashboard.ts`, `data.ts`, `dev.ts`, `listen.ts`, `update.ts`, `autocomplete/index.ts`, `accounts/manage.ts`,
and `mcp/*.ts`.

## Two command shapes

There are two shapes in this directory today, side by side, as the codebase migrates from one to the other:

1. **Legacy generated shape** (most files not yet migrated). Each `run()` inlines the full pipeline: parse
   flags, validate each field, optional dry-run block, build a `FreeClimbApi` instance, define
   `formatOutput`/`normalResponse`/`nextResponse` closures, then call `fcApi.apiCall(method, {params|data},
   normalResponse)`. These files start with the `/* WARNING: This file is automatically generated... */`
   header comment and are regenerated from `generation/commands/main.js`.

2. **Stub-over-executor shape** (`calls/list.ts`, `calls/get.ts`, `sms/send.ts`, `applications/delete.ts`,
   `applications/list.ts`, and any command migrated after them). Each `run()` builds a declarative
   `CommandSpec` (see `../executor.ts`) and calls `runResourceCommand(this, spec)`. The executor owns
   validation, dry-run preview, confirmation prompts, the promise-based API request (`apiRequest`/
   `publicRequest` from `@freeclimb/core`), 204 handling, quiet-id extraction, `--fields` filtering, the JSON
   envelope, human rendering, cursor pagination, and error-code mapping. Static `description`/`examples`/
   `flags`/`args` are unchanged from the legacy shape — only `run()` collapses.

   A future generator should emit commands directly in this stub shape instead of the legacy inlined
   pipeline; see `../executor.ts` for the `CommandSpec` type and the conversion recipe recorded in the
   Phase 4a implementation notes for mapping old generated fields onto spec fields.

## Command Structure (oclif v4)
Each command file:
- Imports `{ Args, Command, Flags } from "@oclif/core"`
- Uses `Flags.boolean()`, `Flags.string()`, `Args.string()` for definitions
- Includes `--json`, `--fields`, `--dry-run` (mutating only) flags
- Legacy shape: uses `await this.parse(ClassName)` directly and validates inline via `validateResourceId()`,
  `rejectControlChars()`. Stub shape: declares `validations` on the `CommandSpec` instead; the executor calls
  `parse()` and runs validations itself.

## Adding a New Command
1. Prefer the stub-over-executor shape for any new resource command: define static
   `description`/`examples`/`flags`/`args`, then build a `CommandSpec` and call `runResourceCommand(this,
   spec)` in `run()`.
2. Verify: `pnpm exec tsc --noEmit`
3. Add tests in `test/commands/`
